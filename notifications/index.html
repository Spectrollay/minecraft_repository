关于版本机制的变更说明 发布: 2024-07-27

随着4.6.5.20版本的发布(这大概率是今年的最后一个版本), 一项新的版本机制已实装至版本库. 为了防止从旧版本升级上来的用户产生困惑以及更好地服务其他依靠本仓库开发的三方开发者, 我们撰写了本变更说明.

面向普通用户

本版本库主要分为发布仓库和测试仓库两大分支. 发布仓库主要用于发布较为稳定的更新, 测试仓库主要用于发布仍在测试中的更新. 在早先的版本中, 你可以同时不受限制地访问发布仓库和测试仓库, 但是从本版本起, 你将只能使用其中的一个仓库.

发布仓库是默认的仓库, 在正常情况下, 你从外部发现的宣传链接应该是指向发布仓库的. 在使用发布仓库后, 你有几率在版本库主页发现邀测弹窗, 这会让你选择是否加入测试仓库进行测试. 当你选择加入测试时会自动重定向至测试仓库. 当然你也可以直接通过邀测链接加入测试仓库, 这两者是等价的. 不过需要注意的是, 加入测试仓库后除非你选择退出测试否则无法访问发布仓库.

测试仓库是不公开的仓库, 仅可以通过邀测弹窗或邀测链接访问. 测试仓库可能并不稳定, 因此如果你没有一定的自主学习和解决问题的能力不建议加入测试. 加入测试仓库后可以参与反馈激励计划(发布仓库无效), 详情请见下一条消息通知. 想要访问发布仓库你需要前往设置页面找到退出测试选项选择离开, 这会带你返回发布仓库.

面向三方开发者

从本版本起, 我们的版本号将遵循版本号规范发布, 具体内容如下:

版本库的版本号有4位, 其中前2位为主版本号, 第3位为次版本号, 第4位为修订号. 以4.6.5.20为例, 4.6为主版本号, 5为次版本号, 20为修订号. 此外, 版本还具有类型/名称等属性, 这里涉及的更改并不多, 不做过多赘述.

主版本号的第1位通常在服务器发生更改时变动, 第2位通常在发布大量新功能/进行大量的改进/产生了向下不兼容的更改时变动. 当这两位版本号发生变动时你需要对仓库中的代码进行重新适配. 次版本号(第3位)通常在加入向下兼容的新内容时变动, 在主版本号变动时会置0或置1. 修订号(第4位)通常在发生向下兼容的问题修正时变动, 次版本号变动时不一定重置. 次版本号和修订号的变动通常可以直接合并, 在没有对相应部分进行更改的情况下不需要进行过多适配.

此外, 发版模式也有所更改. 我们将使用新的优选更新模式取代原先的累积更新模式. 现在发布仓库和测试仓库不再是更新内容不同的分支而是同一更新内容处于不同时间线上的分支了. 测试仓库现在只会更新Preview / Insider_(Preview/Alpha/Beta) / Canary / Alpha这几个版本类型了, 发布仓库现在会更新Beta / Pre / RC / Release / SP这几个版本类型了.

具体发版流程: 新的内容会先在开发仓库进行更新和测试, 等测试周期结束且无问题后才会更新至发布仓库. 开发仓库仍会保持原有的更新频率, 但是不会再更新正式版本了. 开发仓库中的版本在发布后会进行评估, 合格的版本将会优选至发布仓库中进行更新. 这两个版本仅在发布日期和版本类型上存在区别, 其他内容完全一致. 举个例子: 测试仓库发布了1.0.0.1.Preview, 1.0.0.2.Preview, 1.0.0.3.Preview, 1.0.0.4.Preview4个版本, 在分别经过了评估周期后1.0.0.3.Preview被评定为合格版本, 优选至发布仓库成为1.0.0.3.Beta.

这会带来两个好处, 一是不需要再同时维护两套代码了, 现在开发只需要维护发布仓库即可了, 节省了一些不必要的工作量; 二是大大缩短了发布仓库的更新周期, 使得新特性不必再等到下一次正式版本更新才能进版, 可以在一定程度上优化用户的使用体验.

作为三方开发者, 你也应该遵守上面的规范. 此外, 你还需要在正式版本更新的半年内适配对应的版本, 及时对更改做出跟进.

版本库V4正式发布! 发布: 2024-02-09

2023年的10月初, 我们发布了版本库V4的首个预览版. 这是自2021年12月底V3.1.7版本更新之后很长一段时间内的首次更新. 一方面, 是发现很久没更新了, 里面的东西很多都过时了, 另一方面, 是相关联的项目有了具体的推进, 想着顺带对版本库一同进行重构. 于是, 版本库V4就提上日程了.

不同于版本库V1-V3采用的默认HTML样式及后面改善的Minecraft Feedback网站样式, V4采用了全新的OreUI样式(虽然直到现在仍在开发中), 大大提高了界面的美观程度, 当然在另一方面这也带来了大量的设计成本和调试成本, 因此开发进程并不是那么顺利.

经过了4个多月的开发, 迭代了55个开发版本, 直到现在, 版本库V4才达到了我们认为可以公开发布的标准. 因此, 我们于2024年的春节当天, 一个辞旧迎新的好日子, 公开发布了它, 作为早已停止维护的版本库V3的接替. 至此, 版本库V3完成了它的使命正式下线.

诚然, 如版本库首页所言, 我们深知这个版本库还很不尽人意, 界面简陋, 功能稀少, 甚至可能还有一堆的问题. 相较于版本库V3缺少了些许内容. 因此我们仍在不断地完善改进它, 希望能给每一个使用版本库的你, 带来更好的体验. 版本库V4的发布, 不代表开发的告一段落, 而是一个新的开始.

那么, 新的一年, 让我们一起好好加油吧. 在这里也祝大家龙年大吉!
